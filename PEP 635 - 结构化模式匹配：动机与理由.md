## 动机

（结构化）模式匹配语法出现在许多语言中，从Haskell、Erlang和Scala到Elixir和Ruby。（JavaScript的提案也在考虑中。）

Python已经通过序列解包赋值支持了这种功能的有限形式，新提案正是基于此扩展的。

还有其他几个常见的Python习惯用法也与此相关：

- `if ... elif ... elif ... else` 习惯用法经常用于以临时方式找出对象的类型或形状，使用一个或多个检查如 `isinstance(x, cls)`、`hasattr(x, "attr")`、`len(x) == n` 或 `"key" in x` 作为守卫来选择适用的代码块。然后该块可以假设 `x` 支持守卫检查的接口。例如：

```python
if isinstance(x, tuple) and len(x) == 2:
    host, port = x
    mode = "http"
elif isinstance(x, tuple) and len(x) == 3:
    host, port, mode = x
# 等等
```

用 `match` 可以更优雅地呈现这样的代码：

```python
match x:
    case host, port:
        mode = "http"
    case host, port, mode:
        pass
    # 等等
```

- AST遍历代码经常查找匹配给定模式的节点，例如检测形状为"A + B * C"的节点的代码可能如下：

```python
if (isinstance(node, BinOp) and node.op == "+"
        and isinstance(node.right, BinOp) and node.right.op == "*"):
    a, b, c = node.left, node.right.left, node.right.right
    # 处理 a + b*c
```

使用 `match` 这变得更易读：

```python
match node:
    case BinOp("+", a, BinOp("*", b, c)):
        # 处理 a + b*c
```

我们相信，将模式匹配添加到Python将使Python用户能够为上述示例及更多其他情况编写更清晰、更易读的代码。

### 模式匹配与面向对象

模式匹配与面向对象范式是互补的。使用OO和继承，我们可以轻松地在基类上定义方法来定义该类上特定操作的默认行为，并且可以在子类中覆盖此默认行为。我们也可以使用Visitor模式将操作与数据分离。

但这并非对所有情况都足够。例如，代码生成器可能使用AST，并且有许多操作需要生成的代码不仅根据节点的类，还根据某些类属性的值（如上面的 `BinOp` 示例）而变化。Visitor模式对于这种情况不够灵活：它只能基于类进行选择。

与Visitor模式一样，模式匹配允许严格分离关注点：特定操作或数据处理独立于类层次结构或被操作的对象。特别是当处理预定义甚至内置类时，通常不可能向各个类添加更多方法。模式匹配不仅使程序员或类设计者免除了Visitor模式所需的样板代码的负担，而且还足够灵活，可以直接使用内置类型。它自然地区分不同长度的序列，这些序列可能共享同一个类，尽管结构明显不同。此外，模式匹配自动考虑继承：默认情况下，继承自*C*的类*D*将被针对*C*的模式处理。

面向对象编程面向单分派：是一个单独的实例（或其类型）决定调用哪个方法。在二元运算符的情况下，这会导致一个有些人为的情况，即两个对象可能在决定使用哪种实现方面发挥同等作用（Python通过使用反向二元方法来解决这个问题）。模式匹配在结构上更适合处理这种多分派情况，即所采取的动作取决于多个对象的类型，且权重相等。

### 模式与函数式风格

许多Python应用程序和库并非以一致的OO风格编写——与Java不同，Python鼓励在模块的顶层定义函数，对于简单的数据结构，元组（或命名元组或列表）和字典通常单独使用或与类或数据类混合使用。

模式匹配特别适合分解这些数据结构。作为一个极端的例子，很容易用 `match` 编写从JSON数据结构中选取数据的代码：

```python
match json_pet:
    case {"type": "cat", "name": name, "pattern": pattern}:
        return Cat(name, pattern)
    case {"type": "dog", "name": name, "breed": breed}:
        return Dog(name, breed)
    case _:
        raise ValueError("Not a suitable pet")
```

函数式编程通常偏好声明式风格，侧重于数据中的关系。尽可能避免副作用。因此，模式匹配自然适合并高度支持函数式编程风格。

## 理由

本节为各个设计决策提供了理由。它取代了标准PEP格式中的"被否决的想法"部分。它按照规范（[PEP 634](../pep-0634/)）的章节组织。

### 概述与术语

模式匹配的强大之处很大程度上来自子模式的嵌套。因此，模式匹配的成功直接取决于子模式的成功，这是设计的基石。然而，尽管像 `P(Q(), R())` 这样的模式只有在其子模式 `Q()` 和 `R()` 都成功时才成功（即模式 `P` 的成功取决于 `Q` 和 `R`），但模式 `P` 是首先被检查的。如果 `P` 失败，则 `Q()` 和 `R()` 都不会被尝试（这是因为如果 `P` 失败，首先就没有主题来与 `Q()` 和 `R()` 匹配这一事实的直接后果）。

还要注意，模式将名称绑定到值而不是执行赋值。这反映了模式旨在没有副作用这一事实，这也意味着捕获或AS模式不能给属性或下标赋值。因此，我们始终使用术语"绑定"而不是"赋值"，以强调传统赋值与模式中名称绑定之间的微妙差异。

### match语句

match语句评估一个表达式以产生主题，找到与主题匹配的第一个模式，并执行相关的代码块。从语法上讲，match语句因此接受一个表达式和一系列case子句，每个case子句包含一个模式和一个代码块。

由于case子句包含一个代码块，它们遵循现有的缩进方案，句法结构为 `<keyword> ...: <(indented) block>`，类似于复合语句。关键字 `case` 反映了它在模式匹配语言中的广泛使用，忽略了那些使用其他语法手段（如 `|` 等符号）的语言，因为它不符合已建立的Python结构。模式在关键字后面的语法将在下面讨论。

鉴于case子句遵循复合语句的结构，match语句本身自然也成为复合语句，遵循相同的语法结构。这自然导致 `match <expr>: <case_clause>+`。注意，match语句决定了一个准作用域，在其中被评估的主题保持活跃（尽管不在局部变量中），类似于with语句如何在执行其块期间保持资源活跃。此外，控制流从match语句流向case子句，然后离开match语句的块。因此，match语句的块同时具有句法和语义意义。

各种建议试图消除或避免case子句代码块的自然产生的"双重缩进"。不幸的是，所有这些*平面缩进方案*的提议都以违反Python既定结构范式为代价，导致额外的语法规则：

- **不缩进的case子句。** 想法是将case子句与 `match` 对齐，即：

```python
match expression:
case pattern_1:
    ...
case pattern_2:
    ...
```

这在Python程序员的眼中看起来可能很奇怪，因为其他地方的冒号后面都有缩进。`match` 既不遵循简单语句也不遵循复合语句的语法方案，而是建立了自己的类别。

- **将表达式放在"match"后面的单独一行。** 想法是使用产生主题的表达式作为语句，以避免 `match` 没有实际块但有冒号的奇异情况：

```python
match:
    expression
case pattern_1:
    ...
case pattern_2:
    ...
```

这最终被否决了，因为第一个块将是Python语法中的另一个新奇事物：一个只包含单个表达式而不是语句序列的块。试图通过添加或重新利用像 `match: return expression` 这样的另一个关键字来修改这个问题，但没有产生令人满意的解决方案。

尽管平面缩进可以节省一些水平空间，但增加复杂性或非常规规则的代价太高了。这也会使简单的代码编辑器的生活复杂化。最后，水平空间问题可以通过允许match语句使用"半缩进"（即两个空格而不是四个）来缓解（尽管我们不推荐这样做）。

在作为本PEP开发的一部分编写的使用 `match` 的示例程序中，观察到代码简洁性有明显改善，足以弥补额外的缩进级别。

**语句 vs 表达式。** 一些建议集中在使 `match` 成为表达式而不是语句的想法上。然而，这与Python的语句导向性质不太匹配，会导致异常长和复杂的表达式，并且需要发明新的语法构造或破坏已建立的语法规则。作为表达式 `match` 的一个明显后果是，case子句不能再附加任意代码块，而只能附加单个表达式。总体而言，在某些特殊用例中的轻微简化根本无法抵消这些强烈限制。

**硬关键字 vs 软关键字。** 有选项使match成为硬关键字，或选择不同的关键字。尽管使用硬关键字会简化简单语法高亮器的生活，我们决定不使用硬关键字，原因如下：

- 最重要的是，新的解析器不需要我们这样做。与 `async` 在少数版本中作为软关键字造成困难不同，这里我们可以使 `match` 成为永久软关键字。
- `match` 在现有代码中如此常用，以至于它会破坏几乎每个现有程序，并将负担放在可能甚至不受益于新语法的人身上修复代码。
- 很难找到一个不会在现有程序中作为标识符常用但仍能清楚反映语句含义的替代关键字。

**使用"as"或"|"代替"case"作为case子句。** 建议的模式匹配是多分支控制流（与Algol派生语言中的 `switch` 或Lisp中的 `cond` 一致）和函数式语言中的对象解构的组合。虽然建议的关键字 `case` 强调了多分支方面，但替代关键字如 `as` 也同样可能，强调解构方面。例如，`as` 或 `with` 也具有已经是Python关键字的优势。然而，由于 `case` 作为关键字只能作为 `match` 语句内的前导关键字出现，解析器很容易区分其作为关键字或变量的使用。

其他变体将使用 `|` 或 `=>` 这样的符号，或者完全不使用特殊标记。

由于Python是Algol传统的面向语句的语言，并且每个复合语句都以识别关键字开头，`case` 似乎最符合Python的风格和传统。

#### 匹配语义

不同case子句的模式可能重叠，即多个case子句可能匹配给定的主题。先匹配规则确保了对给定主题的case子句选择是明确的。此外，case子句可以有越来越通用的模式匹配更广泛的主题集合。然后，先匹配规则确保可以选择最精确的模式（尽管正确排序case子句是程序员的责任）。

在静态类型语言中，match语句将被编译为决策树，以非常快速和高效地选择匹配的模式。然而，这将要求所有模式都是纯声明性和静态的，与Python既定的动态语义相冲突。因此，建议的语义代表了一条结合两者优点的路径：模式按严格顺序尝试，使每个case子句构成一个实际语句。同时，我们允许解释器缓存关于主题的任何信息或更改尝试子模式的顺序。换句话说：如果解释器发现主题不是类 `C` 的实例，它可以直接跳过再次测试此内容的case子句，而不必执行重复的实例检查。如果守卫规定变量 `x` 必须是正的（即 `if x > 0`），解释器可能在绑定 `x` 之后且在考虑任何进一步子模式之前直接检查此条件。

**绑定和作用域。** 在许多模式匹配实现中，每个case子句都会建立自己的单独作用域。然后，由模式绑定的变量将只在相应case块内可见。然而，在Python中，这没有意义。建立单独的作用域基本上意味着每个case子句是一个单独的函数，无法直接访问周围作用域中的变量（不必求助于 `nonlocal` 就是如此）。此外，case子句不能再通过 `return` 或 `break` 等标准语句影响周围控制流。因此，这种严格的作用域会导致违反直觉和令人惊讶的行为。

直接后果是，任何变量绑定都会延续到各自的case或match语句之后。甚至只部分匹配主题的模式可能会绑定局部变量（这实际上是守卫正常功能所必需的）。然而，这些变量绑定的语义与for循环和with语句等现有Python结构是一致的。

#### 守卫

有些约束不能仅通过模式充分表达。例如，"小于"或"大于"关系与模式的通常"相等"语义相矛盾。此外，不同的子模式是独立的，不能相互引用。添加*守卫*解决了这些限制：守卫是附加到模式的任意表达式，必须评估为"真"值才能使模式成功。

例如，`case [x, y] if x < y:` 使用守卫（`if x < y`）来表达两个原本独立的捕获模式 `x` 和 `y` 之间的"小于"关系。

从概念上讲，模式以声明式风格描述主题上的（结构）约束，理想情况下没有任何副作用。特别要记住，模式明显不同于表达式，遵循不同的目标和语义。然后，守卫以高度受控的方式用任意表达式（可能有副作用）增强case块。将整体功能拆分为静态结构和动态评估部分不仅有助于可读性，而且可以为编译器优化带来巨大的潜力。为了保持这种清晰分离，守卫仅在case子句级别支持，而不是针对单个模式。

**使用守卫的示例：**

```python
def sort(seq):
    match seq:
        case [] | [_]:
            return seq
        case [x, y] if x <= y:
            return seq
        case [x, y]:
            return [y, x]
        case [x, y, z] if x <= y <= z:
            return seq
        case [x, y, z] if x >= y >= z:
            return [z, y, x]
        case [p, *rest]:
            a = sort([x for x in rest if x <= p])
            b = sort([x for x in rest if p < x])
            return a + [p] + b
```

### 模式

模式有两个目的：它们对主题施加（结构）约束，并指定应从主题中提取哪些数据值绑定到变量。在可迭代解包（可以看作是Python中模式匹配的原型）中，只有一个*结构模式*来表达序列，而有一组丰富的*绑定模式*来将值分配给特定的变量或字段。完整的模式匹配与此不同的是，结构模式有更多变化，但绑定模式最少。

模式与赋值目标（如可迭代解包中的）的区别有两个方面：它们对主题结构施加额外约束，主题可能安全地无法匹配特定模式（在可迭代解包中，这构成错误）。后者意味着模式应尽可能避免副作用。

这种避免副作用的愿望是捕获模式不允许绑定值到属性或下标的原因之一：如果包含模式在后续步骤中失败，将很难恢复这些绑定。

模式匹配的一个基石是任意*嵌套模式*的可能性。嵌套允许表达深层树结构（如上面的动机部分中嵌套类模式的示例）以及替代方案。

尽管模式表面上看起来像表达式，但重要的是要记住它们之间有明确的区别。事实上，没有模式是或包含表达式。将模式视为类似于函数定义中的形式参数会更好。

#### AS模式

模式分为两类：大多数模式对主题需要满足的（结构）约束，而捕获模式将主题绑定到名称，不考虑主题的结构或实际值。因此，一个模式可以表达约束或绑定值，但不能同时两者兼有。AS模式填补了这一空白，允许用户指定通用模式以及将主题捕获到变量中。

AS模式的典型用例包括OR和类模式以及绑定名称，如 `case BinOp('+'|'-' as op, ...):` 或 `case [int() as first, int() as second]:`。后者可以被理解为主题必须满足两个不同的模式：`[first, second]` 和 `[int(), int()]`。因此，AS模式可以被视为"与"模式的特例（有关"与"模式的额外讨论，请参见下面的OR模式）。

在早期版本中，AS模式被设计为"海象模式"，写成 `case [first:=int(), second:=int()]`。然而，使用 `as` 比 `:=` 有一些优势：

- 海象运算符 `:=` 用于捕获右侧表达式的结果，而 `as` 通常表示某种形式的"处理"，如 `import foo as bar` 或 `except E as err:`。确实，模式 `P as x` 不会将模式 `P` 赋值给 `x`，而是将成功匹配 `P` 的主题赋值给 `x`。
- `as` 允许更一致地从左到右的数据流（类模式中的属性也遵循从左到右的数据流）。
- 海象运算符看起来与类模式中匹配属性的语法非常相似，可能导致一些混淆。

**使用AS模式的示例：**

```python
def simplify_expr(tokens):
    match tokens:
        case [('('|'[') as l, *expr, (')'|']') as r] if (l+r) in ('()', '[]'):
            return simplify_expr(expr)
        case [0, ('+'|'-') as op, right]:
            return UnaryOp(op, right)
        case [(int() | float() as left) | Num(left), '+', (int() | float() as right) | Num(right)]:
            return Num(left + right)
        case [(int() | float()) as value]:
            return Num(value)
```

#### OR模式

OR模式允许你将"结构等价"的替代项组合成新模式，即多个模式可以共享一个通用处理程序。如果OR模式的任何子模式匹配主题，则整个OR模式成功。

静态类型语言禁止在OR模式内绑定名称（捕获模式），因为变量类型可能存在潜在冲突。作为动态类型语言，Python可以在这里不那么严格，允许OR模式内的捕获模式。但是，每个子模式必须绑定相同的变量集，以免留下可能未定义的名称。对于两个替代项 `P | Q`，这意味着如果 *P* 绑定变量 *u* 和 *v*，则 *Q* 必须恰好绑定相同的变量 *u* 和 *v*。

关于使用竖线符号 `|` 还是 `or` 关键字来分隔替代项进行了一些讨论。OR模式并不完全适合这两个符号中任何一个的现有语义和用法。但是，`|` 是所有支持OR模式的编程语言中选择使用的符号，并且在Python的正则表达式中也以该能力使用。它也是形式语法（包括Python的）中替代项之间的传统分隔符。此外，`|` 不仅用于按位OR，还用于集合并集和字典合并（[PEP 584](../pep-0584/)）。

也考虑了其他替代方案，但这些方案都不能允许OR模式嵌套在其他模式内：

- **使用逗号：**
```python
case 401, 403, 404:
    print("Some HTTP error")
```
这看起来太像元组了——我们必须找到不同的方式来写元组，并且该构造必须在类模式的参数列表内用括号括起来。一般来说，逗号在Python中已经有很多不同的含义，我们不应该添加更多。

- **使用堆叠的case：**
```python
case 401:
case 403:
case 404:
    print("Some HTTP error")
```
这是C语言中的做法，使用其case的贯穿语义。然而，我们不想误导人们认为match/case使用贯穿语义（这是C中常见的错误来源）。此外，这将是一种新奇的缩进模式，可能会使IDE等支持更加困难（它会破坏"在以冒号结尾的行后添加一个缩进级别"这个简单规则）。最后，这也不能支持嵌套在其他模式内的OR模式。

- **使用"case in"后跟逗号分隔的列表：**
```python
case in 401, 403, 404:
    print("Some HTTP error")
```
这不能用于嵌套在其他模式内的OR模式，如：
```python
case Point(0|1, 0|1):
    print("A corner of the unit square")
```

**AND和NOT模式**

既然本提案定义了OR模式（`|`）来匹配多个替代项之一，为什么不也定义AND模式（`&`）甚至NOT模式（`!`）呢？尤其是考虑到其他一些语言（例如F#）支持AND模式。

然而，目前尚不清楚这会有多有用。匹配字典、对象和序列的语义已经包含隐式的"与"：所有提到的属性和元素都必须存在才能使匹配成功。守卫条件也可以支持许多假设的"与"运算符将被用于的用例。

使用运算符 `!` 作为前缀的模式匹配否定会在模式本身不匹配时完全匹配。例如，`!(3 | 4)` 将匹配除 `3` 或 `4` 之外的任何内容。然而，有来自其他语言的证据表明这很少有用，主要用于双重否定 `!!` 来控制变量范围和防止变量绑定（这不适用于Python）。其他用例最好用守卫来表达。

最终，结论是这会使语法更复杂而不增加显著的好处。它可以稍后添加。

**使用OR模式的示例：**

```python
def simplify(expr):
    match expr:
        case ('/', 0, 0):
            return expr
        case ('*'|'/', 0, _):
            return 0
        case ('+'|'-', x, 0) | ('+', 0, x) | ('*', 1, x) | ('*'|'/', x, 1):
            return x
    return expr
```

#### 字面量模式

字面量模式是对主题的值而不是其类型或结构施加约束的便捷方式。它们还允许你使用模式匹配来模拟switch语句。

通常，主题与字面量模式的比较通过标准相等性（Python语法中的 `x == y`）进行。因此，字面量模式 `1.0` 和 `1` 匹配完全相同的一组对象，即 `case 1.0:` 和 `case 1:` 是完全可以互换的。原则上，`True` 也会匹配同一组对象，因为 `True == 1` 成立。然而，我们相信许多用户会惊讶地发现 `case True:` 匹配主题 `1.0`，从而导致一些微妙的错误和迂回的解决方法。因此，我们采用规则：三个单例模式 `None`、`False` 和 `True` 通过身份（Python语法中的 `x is y`）而不是相等性进行比较。因此，`case True:` 将只匹配 `True` 而不匹配其他任何东西。注意，`case 1:` 仍然会匹配 `True`，因为字面量模式 `1` 通过相等性而不是身份工作。

早期关于在数字上引入层次结构的的想法被放弃了，取而代之的是基于相等性的更简单和更一致的规则。此外，对 `numbers.Integral` 实例的任何额外检查都会以高昂的运行时成本引入本质上新颖的想法。如有需要，可以使用显式语法 `case int(1):`。

回想一下，字面量模式*不是*表达式，而是直接表示一个特定的值。从实用的角度来看，我们希望允许使用负数甚至复数值作为字面量模式，但它们不是原子字面量（只有无符号实数和虚数才是）。例如，`-3+4j` 语法上是形式为 `BinOp(UnaryOp('-', 3), '+', 4j)` 的表达式。由于表达式不是模式的一部分，我们必须添加显式语法支持这些值，而不必求助于完整表达式。

插值*f*字符串另一方面，虽然外观相似，但不是字面量值，因此不能用作字面量模式（但是支持字符串连接）。

字面量模式不仅作为模式出现，而且也作为*映射模式*中的键出现。

**范围匹配模式。** 这将允许像 `1...6` 这样的模式。然而，存在很多歧义：

- 范围是开放的、半开放的还是封闭的？（即在上面的例子中 `6` 是否包含？）
- 范围匹配单个数字，还是范围对象？
- 范围匹配通常用于字符范围（'a'...'z'），但在Python中这不起作用，因为没有字符数据类型，只有字符串。
- 如果可以预先构建跳转表，范围匹配可以是一个重要的性能优化，但在Python中通常不可能，因为名称可以动态重新绑定。

而不是为范围创建特殊的语法，决定允许自定义模式对象（`InRange(0, 6)`）会更灵活和更少歧义；但是这些想法暂时被推迟了。

**使用字面量模式的示例：**

```python
def simplify(expr):
    match expr:
        case ('+', 0, x):
            return x
        case ('+' | '-', x, 0):
            return x
        case ('and', True, x):
            return x
        case ('and', False, x):
            return False
        case ('or', False, x):
            return x
        case ('or', True, x):
            return True
        case ('not', ('not', x)):
            return x
    return expr
```

#### 捕获模式

捕获模式采用名称的形式，接受任何值并将其绑定到（局部）变量（除非该名称声明为 `nonlocal` 或 `global`）。从这个意义上说，捕获模式类似于函数定义中的参数（当调用函数时，每个参数将相应的参数绑定到函数作用域中的局部变量）。

用于捕获模式的名称不能与同一模式中的另一个捕获模式重合。这再次类似于参数，参数同样要求每个参数名称在参数列表中是唯一的。然而，它与可迭代解包赋值不同，在可迭代解包赋值中，重复使用变量名作为目标是允许的（例如 `x, x = 1, 2`）。不支持模式中 `(x, x)` 的理由是其阅读是模糊的：它可以被视为可迭代解包，其中只有第二个绑定到 `x` 保留下来。但它同样可以被视为表达具有两个相等元素的元组（这本身也有问题）。如果需要，仍然可以稍后引入对重复使用名称的支持。

有人呼吁显式标记捕获模式，从而将它们识别为绑定目标。根据该想法，捕获模式将被写成例如 `?x`、`$x` 或 `=x`。这种显式捕获标记的目的是让未标记的名称成为值模式（见下文）。然而，这是基于对模式匹配是*switch*语句扩展的误解，强调基于（序数）值的快速切换。这样的*switch*语句之前确实已经为Python提出过（见 [PEP 275](../pep-0275/) 和 [PEP 3103](../pep-3103/)）。另一方面，模式匹配构建了可迭代解包的广义概念。从数据结构中提取绑定值是概念的核心，因此是最常见的用例。显式捕获标记会背叛提议的模式匹配语法的目标，并以核心用例的额外语法混乱为代价简化次要用例。

有人提出捕获模式根本不需要，因为可以通过将AS模式与通配符模式组合来获得等效效果（例如 `case _ as x` 等效于 `case x`）。然而，这会令人不快地冗长，特别是考虑到我们期望捕获模式非常常见。

**使用捕获模式的示例：**

```python
def average(*args):
    match args:
        case [x, y]:           # 捕获序列的两个元素
            return (x + y) / 2
        case [x]:              # 捕获序列的唯一元素
            return x
        case []:
            return 0
        case a:                # 捕获整个序列
            return sum(a) / len(a)
```

#### 通配符模式

通配符模式是"捕获"模式的一种特殊情况：它接受任何值，但不将其绑定到变量。这条规则背后的想法是支持在模式中重复使用通配符。虽然 `(x, x)` 是错误，但 `(_, _)` 是合法的。

特别是在较大的（序列）模式中，重要的是允许模式专注于具有实际意义的值而忽略其他任何内容。没有通配符，就有必要"发明"一些局部变量，这些变量会被绑定但从未被使用。即使坚持命名约定并使用例如 `_1, _2, _3` 来命名无关的值，这仍然会引入视觉混乱并可能影响性能（比较序列模式 `(x, y, *z)` 和 `(_, y, *_)`，其中 `*z` 强制解释器复制可能很长的序列，而第二个版本只是编译为类似于 `y = seq[1]` 的代码）。

关于选择下划线 `_` 作为通配符模式进行了很多讨论，即让这一个名称不绑定。然而，下划线已经在可迭代解包中大量用作"忽略值"标记。由于通配符模式 `_` 从不绑定，这一下划线的使用不会干扰其他使用，如REPL内或gettext模块中。

有人提议使用 `...`（即省略号标记）或 `*`（星号）作为通配符。然而，两者看起来都像是省略了任意数量的项目：

```python
case [a, ..., z]: ...
case [a, *, z]: ...
```

任一示例看起来都像它会匹配两个或更多项目的序列，捕获第一个和最后一个值。虽然这可能是最终的"通配符"，但它没有传达所需的语义。

一个不暗示任意数量项目的替代方案是 `?`。这甚至在 [PEP 640](../pep-0640/) 中独立于模式匹配提出。然而，我们认为使用 `?` 作为特殊的"赋值"目标可能比使用 `_` 更让Python用户困惑。它违反了Python（诚然模糊的）原则：只在类似于普通英语用法或高中数学中使用标点符号，除非用法在其他编程语言中*非常*确立（例如使用点进行成员访问）。

问号在两个方面都失败了：它在其他编程语言中的用法是一个杂乱的用法集，只是模糊地由"问题"的想法暗示。例如，它在shell globbing中表示"任何字符"，在正则表达式中表示"可能"，在C和许多C派生语言中表示"条件表达式"，在Scheme中表示"谓词函数"，在Rust中表示"修改错误处理"，在TypeScript中表示"可选参数"和"可选链"（后者的含义也被 [PEP 505](../pep-0505/) 建议用于Python）。一个尚未命名的PEP建议用它来标记可选类型，例如 `int?`。

编程系统中 `?` 的另一个常见用途是"帮助"，例如在IPython和Jupyter Notebook以及许多交互式命令行工具中。

此外，这将使Python处于相当独特的地位：在我们能找到的所有支持模式匹配的编程语言中（包括C#、Elixir、Erlang、F#、Grace、Haskell、Mathematica、OCaml、Ruby、Rust、Scala、Swift和Thorn），下划线都是通配符模式。考虑到许多Python用户也使用其他编程语言，在学习Python之前有先验经验，并且可能在学习Python之后转向其他语言，我们发现这种既定标准对可读性和可学习性很重要。在我们看来，担心这个通配符意味着常规名称获得了特殊待遇的说法不够有力，不足以引入使Python特殊的语法。

**else块。** 没有守卫且模式为单个通配符的case块（即 `case _:`）接受任何主题而不将其绑定到变量或执行任何其他操作。因此它在语义上等同于 `else:`（如果支持的话）。但是，将这样的else块添加到match语句语法中不会消除在其他上下文中对通配符模式的需求。另一个反对意见是，else块有两种合理的缩进级别：与 `case` 对齐或与 `match` 对齐。作者们发现哪个缩进级别更可取是有争议的。

**使用通配符模式的示例：**

```python
def is_closed(sequence):
    match sequence:
        case [_]:               # 任何具有单个元素的序列
            return True
        case [start, *_, end]:  # 至少有两个元素的序列
            return start == end
        case _:                 # 任何
            return False
```

#### 值模式

使用命名常量来表示参数化值或阐明特定值的含义是良好的编程风格。显然，编写 `case (HttpStatus.OK, body):` 比 `case (200, body):` 更可取。这里出现的主要问题是如何区分捕获模式（变量绑定）和值模式。围绕这个问题的普遍讨论提出了大量选项，我们无法在此全部列出。

严格来说，值模式并不是真正必要的，但可以使用守卫来实现，即 `case (status, body) if status == HttpStatus.OK:`。尽管如此，值模式的便利性是毫无疑问和明显的。

观察到常量倾向于用大写字母书写或收集在类似枚举的命名空间中，这提示了可能的规则来句法上区分常量。然而，使用大写/小写作为标记的想法受到了质疑，因为核心Python中没有类似的先例（尽管在其他语言中很常见）。因此，我们只采用规则：任何带点号的名字（即属性访问）将被解释为值模式，例如上面的 `HttpStatus.OK`。这排除了本地变量和当前模块中定义的全局变量作为常量。

建议使用前导点（例如 `.CONSTANT`）的规则被批评了，因为人们认为点对于该目的来说不是一个足够明显的标记。受其他编程语言形式的启发，提出了许多不同的标记/符号（如 `^CONSTANT`、`$CONSTANT`、`==CONSTANT`、`CONSTANT?` 或用反引号括起来的单词），尽管没有明显或自然的选择。因此，当前提案将此"常量"标记的讨论和可能引入留给未来的PEP。

基于名称是否是全局变量（即编译器将全局变量视为常量而不是捕获模式）来区分名称的语义会导致各种问题。模块中添加或更改全局变量可能对模式产生意想不到的副作用。此外，模式匹配不能直接在模块作用域中使用，因为所有变量都是全局的，使得捕获模式不可能。

**使用值模式的示例：**

```python
def handle_reply(reply):
    match reply:
        case (HttpStatus.OK, MimeType.TEXT, body):
            process_text(body)
        case (HttpStatus.OK, MimeType.APPL_ZIP, body):
            text = deflate(body)
            process_text(text)
        case (HttpStatus.MOVED_PERMANENTLY, new_URI):
            resend_request(new_URI)
        case (HttpStatus.NOT_FOUND):
            raise ResourceNotFound()
```

#### 分组模式

允许用户显式指定分组在OR模式的情况下特别有帮助。

#### 序列模式

序列模式尽可能遵循已建立的可迭代解包的语法和语义。当然，子模式代替赋值目标（变量、属性和下标）。此外，序列模式只匹配精心选择的可能主题集合，而可迭代解包可以应用于任何可迭代对象。

- 与可迭代解包一样，我们不区分"元组"和"列表"表示法。`[a, b, c]`、`(a, b, c)` 和 `a, b, c` 都是等价的。虽然这意味着我们有冗余表示法，专门检查列表或元组需要更多努力（例如 `case list([a, b, c])`），但我们尽可能模仿可迭代解包。
- 星号模式将捕获任意长度的子序列，再次反映可迭代解包。任何序列模式中只能存在一个带星号的项目。理论上，像 `(*_, 3, *_)` 这样的模式可以被理解为表达包含值 `3` 的任何序列。然而，在实践中，这只适用于非常狭窄的用例集，否则会导致低效的回溯甚至歧义。
- 序列模式*不会*遍历可迭代主题。所有元素都通过下标和切片访问，主题必须是 `collections.abc.Sequence` 的实例。这当然包括列表和元组，但例如不包括集合和字典。虽然它会包括字符串和字节，但我们为这些做了例外（见下文）。

序列模式不能只是遍历任何可迭代对象。如果整体模式失败，必须撤消对迭代中元素的消耗，这不可行。

为了识别序列，我们不能仅依靠 `len()` 和下标/切片，因为序列在这些方面与映射（例如 `dict`）共享协议。如果序列模式也匹配字典或其他实现映射协议的对象（即 `__getitem__`），那将是令人惊讶的。因此，解释器执行实例检查以确保相关主题确实是序列（已知类型）。（作为最常见情况的优化，如果主题恰好是列表或元组，可以跳过实例检查。）

字符串和字节对象具有双重性质：它们既是自己本身的"原子"对象，也是序列（具有强烈递归性质，因为字符串是字符串的序列）。字符串和字节的典型行为和用例与元组和列表的典型行为和用例足够不同，值得明确区分。实际上，字符串作为序列通过是经常令人惊讶和非预期的，有规律的问题和抱怨为证。因此，字符串和字节不被序列模式匹配，将序列模式限制为"序列"的非常具体的理解。内置 `bytearray` 类型作为 `bytes` 的可变版本，也值得例外；但我们不打算列举所有其他可能用于表示字节的类型（例如 `memoryview` 和 `array.array` 的某些但不是全部实例）。

#### 映射模式

字典或映射是Python中最重要和最广泛使用的数据结构之一。与序列相反，映射用于通过键快速直接访问任意元素。在大多数情况下，元素是通过已知键从字典中检索的，不考虑存储在同一字典中的任何顺序或其他键值对。字符串键尤其常见。

映射模式反映了字典查找的常见用法：它允许用户通过常量/已知键从映射中提取一些值，并让值匹配给定的子模式。主题中的额外键会被忽略，即使没有 `**rest`。这与序列模式不同，在序列模式中，额外项目会导致匹配失败。但映射实际上与序列不同：它们具有自然的结构子类型行为，即传递带有额外键的字典到某处可能只是工作。如果需要为映射设置上限并确保没有其他键存在，则可以使用通常的双星模式 `**rest`。但是，带通配符的特殊情况 `**_` 不受支持，因为它不会有任何效果，但可能导致对映射模式语义的错误理解。

为避免过度昂贵的匹配算法，键必须是字面量或值模式。

使用 `get(key, default)` 而不是 `__getitem__(key)` 后跟 `AttributeError` 检查有一个微妙的原因：如果主题恰好是 `defaultdict`，调用不存在键的 `__getitem__` 会添加该键。使用 `get()` 可以避免这种意外的副作用。

**使用映射模式的示例：**

```python
def change_red_to_blue(json_obj):
    match json_obj:
        case { 'color': ('red' | '#FF0000') }:
            json_obj['color'] = 'blue'
        case { 'children': children }:
            for child in children:
                change_red_to_blue(child)
```

#### 类模式

类模式有两个目的：检查给定主题是否确实是特定类的实例，并从主题的特定属性中提取数据。有趣的证据表明 `isinstance()` 是Python程序中静态出现最频繁的函数之一。这种实例检查通常先于对存储在对象中的信息的随后访问或其可能的操作。一个典型模式可能如下：

```python
def traverse_tree(node):
    if isinstance(node, Node):
        traverse_tree(node.left)
        traverse_tree(node.right)
    elif isinstance(node, Leaf):
        print(node.value)
```

在许多情况下，类模式是嵌套出现的，如动机部分给出的示例：

```python
if (isinstance(node, BinOp) and node.op == "+"
        and isinstance(node.right, BinOp) and node.right.op == "*"):
    a, b, c = node.left, node.right.left, node.right.right
    # 处理 a + b*c
```

类模式让你简洁地指定实例检查以及相关属性（可能有进一步约束）。因此，很容易写出例如第一个情况中的 `case Node(left, right):` 和第二个情况中的 `case Leaf(value):`。虽然这在具有严格代数数据类型的语言中效果很好，但对于Python对象的结构来说是有问题的。

在处理通用Python对象时，我们面临大量无序属性的潜在问题：`Node` 的实例包含大量属性（其中大多数是"特殊方法"，如 `__repr__`）。此外，解释器无法可靠地推断属性的顺序。对于表示圆的对象，例如，属性 `x`、`y` 和 `radius` 没有固有的明显顺序。

我们设想两种处理此问题的方法：要么显式命名感兴趣的属性，要么提供额外的映射告诉解释器要提取哪些属性以及按什么顺序。两种方法都支持。此外，显式命名感兴趣的属性让你进一步指定对象的必需结构；如果对象缺少模式指定的属性，匹配失败。

- 显式命名的属性采用命名参数的语法。如果 `Node` 类的对象有两个属性 `left` 和 `right` 如上，模式 `Node(left=x, right=y)` 将提取两个属性的值并分别赋值给 `x` 和 `y`。从左到右的数据流看起来不寻常，但与映射模式一致，并且在 *with* 或 *import* 语句中通过 `as` 进行赋值等有先例（实际上是AS模式）。

显式命名相关属性主要用于比位置形式（见下文）更复杂的用例不足的情况。

- 类字段 `__match_args__` 指定一组属性及其顺序，允许类模式依赖位置子模式，而不必显式命名相关属性。这对于较小的对象或数据类实例特别方便，其中感兴趣的属性相当明显，通常具有明确定义的顺序。在某种程度上，`__match_args__` 类似于形式参数的声明，允许使用位置参数而不是命名所有参数来调用函数。

这是一个类属性，因为它需要在类模式中命名的类上查找，而不是在主题实例上。

类模式的语法基于解构镜像构造语法的想法。这已经在 virtually 任何 Python 构造中都是这种情况，无论是赋值目标、函数定义还是可迭代解包。在所有这些情况下，我们发现发送和接收"数据"的语法几乎是相同的。

- 赋值目标如变量、属性和下标：`foo.bar[2] = foo.bar[3]`;
- 函数定义：用 `def foo(x, y, z=6)` 定义的函数可以如 `foo(123, y=45)` 调用，其中在调用站点提供的实际参数与在定义站点的形式参数匹配；
- 可迭代解包：`a, b = b, a` 或 `[a, b] = [b, a]` 或 `(a, b) = (b, a)`，仅举几个等效可能性。

使用相同的语法进行读写、左值和右值，或构造和解构，被广泛接受是因为它在思考数据、其流动和操作方面的好处。这同样延伸到实例的显式构造，其中类模式 `C(p, q)` 故意镜像创建实例的语法。

对于内置类如 `bool`、`bytearray` 等的特殊情况（例如 `str(x)` 在 `x` 中捕获主题值），可以通过用户定义的类来模拟，如下所示：

```python
class MyClass:
    __match_args__ = ["__myself__"]
    __myself__ = property(lambda self: self)
```

**模式变量的类型注解。** 提案是将模式与类型注解结合：

```python
match x:
    case [a: int, b: str]: print(f"An int {a} and a string {b}:")
    case [a: int, b: int, c: int]: print("Three ints", a, b, c)
    ...
```

这个想法有很多问题。一方面，冒号只能在括号或括号内使用，否则语法会变得模糊。而且因为Python不允许对泛型进行 `isinstance()` 检查，包含泛型的类型注解将无法按预期工作。

## 历史与背景

模式匹配于1970年代末以元组解包的形式出现，作为处理递归数据结构（如链表或树）的一种手段（面向对象语言通常使用visitor模式来处理递归数据结构）。模式匹配的早期支持者将结构化数据组织成"标记元组"，而不是C中的 `struct` 或后来引入的对象。例如，二叉树中的节点将是一个带有左右分支两个元素的元组，以及一个 `Node` 标签，写成 `Node(left, right)`。在Python中，我们可能会将标签放在元组内为 `('Node', left, right)` 或定义一个数据类 `Node` 来达到相同的效果。

使用现代语法，深度优先树遍历可以这样写：

```python
def traverse(node):
    match node:
        case Node(left, right):
            traverse(left)
            traverse(right)
        case Leaf(value):
            handle(value)
```

用模式匹配处理递归数据结构的概念立即引发了用模式匹配处理更一般的递归"模式"（即超越递归数据结构的递归）的想法。因此，模式匹配也将用于定义递归函数，例如：

```python
def fib(arg):
    match arg:
        case 0:
            return 1
        case 1:
            return 1
        case n:
            return fib(n-1) + fib(n-2)
```

随着模式匹配反复被集成到新的和新兴的编程语言中，其语法略有发展和扩展。`fib` 示例中的前两个情况可以更简洁地写成 `case 0 | 1:`，其中 `|` 表示替代模式。此外，下划线 `_` 被广泛采用为通配符，一个填充符，在模式的任何部分的结构和值都不重要的地方。由于下划线已经经常以同等能力在Python的可迭代解包中使用（例如 `_, _, third, _* = something`），我们保留了这些通用标准。

值得注意的是，模式匹配的概念一直与函数的概念密切相关。不同的case子句一直被认为是类似于半独立函数的东西，其中模式变量承担参数的角色。当模式匹配被写成重载函数时，这一点最为明显（标准ML）：

```python
fun fib 0 = 1
  | fib 1 = 1
  | fib n = fib (n-1) + fib (n-2)
```

即使在Python中这种case子句严格分离为独立函数的规则不适用，我们发现模式与参数共享许多语法规则，例如只将参数绑定到非限定名称，或者特定模式/函数的变量/参数名称不能重复。

强调抽象和封装的面向对象编程对模式匹配提出了严峻挑战。简言之：在面向对象编程中，我们不能再将对象视为标记元组。传入构造函数的参数不一定指定对象的属性或字段。此外，对象的字段没有严格的顺序，其中一些字段可能是私有的，因此无法访问。最重要的是，给定对象可能实际上是结构稍有不同的子类的实例。

为了应对这一挑战，模式变得越来越独立于原始的元组构造函数。在像 `Node(left, right)` 这样的模式中，`Node` 不再是一个被动标签，而是一个可以主动检查任何给定对象是否具有正确结构并提取 `left` 和 `right` 字段的函数。换句话说：`Node` 标签变成了一个将对象转换为元组的函数，或者在不可能时返回某种失败指示器。

在Python中，我们简单地使用 `isinstance()` 以及类的 `__match_args__` 字段来检查对象是否具有正确的结构，然后将某些属性转换为元组。例如，对于上面的 `Node` 示例，我们会有 `__match_args__ = ('left', 'right')` 来指示应提取这两个属性以形成元组。也就是说，`case Node(x, y)` 首先会检查给定对象是否是 `Node` 的实例，然后分别将 `left` 赋值给 `x` 和 `right` 赋值给 `y`。

然而，鉴于Python通过"鸭子类型"的动态特性，我们也添加了更直接的方式来指定特定属性的存在或约束。而不是 `Node(x, y)`，你也可以写 `object(left=x, right=y)`，有效地消除 `isinstance()` 检查，从而支持任何具有 `left` 和 `right` 属性的对象。或者你可以结合这些想法写 `Node(right=y)`，以要求是 `Node` 的实例但只提取 `right` 属性的值。
