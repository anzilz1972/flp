## 语法与语义

完整语法见[附录A](#附录-a--完整语法)。

### 概述与术语

模式匹配过程以模式（跟在 `case` 后面）和主题值（跟在 `match` 后面）作为输入。描述该过程的短语包括"模式与主题值匹配"和"我们用模式匹配主题值"。

模式匹配的主要结果是成功或失败。在成功的情况下，我们可以说"模式成功"、"匹配成功"或"模式匹配主题值"。

在许多情况下，模式包含子模式，成功或失败由这些子模式相对于值匹配的成功或失败决定（例如，对于OR模式），或相对于值的部分匹配决定（例如，对于序列模式）。此过程通常从左到右处理子模式，直到确定整体结果。例如，OR模式在第一个成功的子模式处成功，而序列模式在第一个失败的子模式处失败。

模式匹配的次要结果可能是一个或多个名称绑定。我们可以说"模式将值绑定到名称"。当子模式尝试直到第一次成功时，只有成功子模式的绑定有效；当尝试直到第一次失败时，绑定被合并。以下解释了几条更多规则适用于这些情况。

### match语句

语法：

```
match_stmt: "match" subject_expr ':' NEWLINE INDENT case_block+ DEDENT
subject_expr:
    | star_named_expression ',' star_named_expressions?
    | named_expression
case_block: "case" patterns [guard] ':' block
guard: 'if' named_expression
```

规则 `star_named_expression`、`star_named_expressions`、`named_expression` 和 `block` 是[标准Python语法](https://docs.python.org/3.10/reference/grammar.html)的一部分。

规则 `patterns` 在下面指定。

作为上下文，`match_stmt` 是 `compound_statement` 的新替代项：

```
compound_statement:
    | if_stmt
    ...
    | match_stmt
```

关键字 `match` 和 `case` 是软关键字，即它们在其他语法上下文中不是保留字（包括在没有预期冒号的行首时）。这意味着它们仅在作为match语句或case块的一部分时被识别为关键字，允许在所有其他上下文中用作变量或参数名。

#### 匹配语义

match语句首先评估主题表达式。如果存在逗号，则使用标准规则构造元组。

然后使用生成的主题值来选择第一个其模式成功匹配*并且*其守卫条件（如果存在）为"真"的case块。如果没有case块符合条件，则match语句完成；否则，执行所选case块的块。执行嵌套在复合语句中的块的通常规则适用（例如 `if` 语句）。

在成功的模式匹配期间进行的名称绑定会在执行的块之后继续存在，并且可以在match语句之后使用。

在失败的模式匹配期间，某些子模式可能成功。例如，当用值 `[0, 1, 2]` 匹配模式 `(0, x, 1)` 时，如果列表元素从左到右匹配，子模式 `x` 可能成功。实现可以选择为这些部分匹配创建持久绑定或不创建。用户代码（包括match语句）不应依赖于为失败匹配创建的绑定，但也不应假设变量因失败匹配而保持不变。这部分行为故意不指定，以便不同的实现可以添加优化，并防止引入可能限制此功能可扩展性的语义限制。

精确的模式绑定规则因模式类型而异，并在下面详细说明。

#### 守卫

如果case块上有守卫，一旦case块中的模式成功匹配，就会评估守卫中的表达式。如果引发异常，异常会向上冒泡。否则，如果条件为"真"，则选择该case块；如果为"假"，则不选择该case块。

由于守卫是表达式，它们允许有副作用。守卫评估必须从第一个到最后一个case块依次进行，一次一个，跳过其模式不完全成功的case块。（即，即使确定这些模式是否成功可能乱序发生，守卫评估必须按顺序进行。）一旦选择了case块，守卫评估必须停止。

#### 不可反驳的case块

如果我们可以仅从其语法证明模式总是成功，则该模式被认为是不可反驳的。特别是，捕获模式和通配符模式是不可反驳的，左侧不可反驳的AS模式、包含至少一个不可反驳模式的OR模式，以及带括号的不可反驳模式也是如此。

如果case块没有守卫且其模式是不可反驳的，则该case块被认为是不可反驳的。

一个match语句最多只能有一个不可反驳的case块，而且它必须在最后。

### 模式

模式的顶级语法如下：

```
patterns: open_sequence_pattern | pattern
pattern: as_pattern | or_pattern
as_pattern: or_pattern 'as' capture_pattern
or_pattern: '|'.closed_pattern+
closed_pattern:
    | literal_pattern
    | capture_pattern
    | wildcard_pattern
    | value_pattern
    | group_pattern
    | sequence_pattern
    | mapping_pattern
    | class_pattern
```

#### AS模式

语法：

```
as_pattern: or_pattern 'as' capture_pattern
```

（注意：右侧的名称不能是 `_`。）

AS模式将 `as` 关键字左侧的OR模式与主题匹配。如果失败，AS模式失败。否则，AS模式将主题绑定到 `as` 关键字右侧的名称并成功。

#### OR模式

语法：

```
or_pattern: '|'.closed_pattern+
```

当两个或多个模式用竖线（`|`）分隔时，这称为OR模式。（单个闭合模式就是那样。）

只有最后一个子模式可以是不可反驳的。

每个子模式必须绑定相同的名称集。

OR模式依次将每个子模式与主题匹配，直到其中一个成功。然后认为OR模式成功。如果所有子模式都失败，则OR模式失败。

#### 字面量模式

语法：

```
literal_pattern:
    | signed_number
    | signed_number '+' NUMBER
    | signed_number '-' NUMBER
    | strings
    | 'None'
    | 'True'
    | 'False'
signed_number: NUMBER | '-' NUMBER
```

规则 `strings` 和标记 `NUMBER` 在标准Python语法中定义。

支持三引号字符串。支持原始字符串和字节字符串。不支持f字符串。

表单 `signed_number '+' NUMBER` 和 `signed_number '-' NUMBER` 仅在表达复数时允许；它们需要在左侧有一个实数，右侧有一个虚数。

如果主题值使用以下比较规则与字面量表达的值比较相等，则字面量模式成功：

- 数字和字符串使用 `==` 运算符比较。
- 单例字面量 `None`、`True` 和 `False` 使用 `is` 运算符比较。

#### 捕获模式

语法：

```
capture_pattern: !"_" NAME
```

单个下划线（`_`）不是捕获模式（这就是 `!"_"` 所表达的）。它被当作[通配符模式](#id3)。

捕获模式总是成功。它使用为海象运算符在 [PEP 572](../pep-0572/) 中建立的名字绑定作用域规则将主题值绑定到名称。（概要：名称成为最近包含函数作用域中的局部变量，除非有适用的 `nonlocal` 或 `global` 语句。）

在给定的模式中，给定名称只能绑定一次。这禁止例如 `case x, x: ...` 但允许 `case [x] | x: ...`。

#### 通配符模式

语法：

```
wildcard_pattern: "_"
```

通配符模式总是成功。它不绑定任何名称。

#### 值模式

语法：

```
value_pattern: attr
attr: name_or_attr '.' NAME
name_or_attr: attr | NAME
```

模式中的带点名称使用标准Python名称解析规则查找。但是，当相同的值模式在同一个match语句中出现多次时，解释器可以缓存找到的第一个值并重用它，而不是重复相同的查找。（要澄清的是，此缓存严格绑定到给定match语句的给定执行。）

如果找到的值使用 `==` 运算符与主题值比较相等，则模式成功。

#### 分组模式

语法：

```
group_pattern: '(' pattern ')'
```

（关于 `pattern` 的语法，请参见上面的模式。注意它不包含逗号——带括号的系列项目（至少包含一个逗号）是序列模式，`()` 也是。）

带括号的模式没有额外的语法。它允许用户为模式添加括号以强调预期的分组。

#### 序列模式

语法：

```
sequence_pattern:
  | '[' [maybe_sequence_pattern] ']'
  | '(' [open_sequence_pattern] ')'
open_sequence_pattern: maybe_star_pattern ',' [maybe_sequence_pattern]
maybe_sequence_pattern: ','.maybe_star_pattern+ ','?
maybe_star_pattern: star_pattern | pattern
star_pattern: '*' (capture_pattern | wildcard_pattern)
```

（注意，不带尾随逗号的单个带括号模式是分组模式，而不是序列模式。但是，用 `[...]` 括起来的单个模式仍然是序列模式。）

使用 `[...]` 的序列模式、使用 `(...)` 的序列模式和开放序列模式之间没有语义差异。

序列模式最多可以包含一个星号子模式。星号子模式可以出现在任何位置。如果没有星号子模式，则序列模式是固定长度序列模式；否则它是可变长度序列模式。

对于序列模式要成功，主题必须是序列，其中序列的定义是其类为以下之一：

- 从 `collections.abc.Sequence` 继承的类
- 已注册为 `collections.abc.Sequence` 的Python类
- 具有 `Py_TPFLAGS_SEQUENCE` 位集的内置类
- 从上述任何类继承的类（包括在父类的 `Sequence` 注册之前定义的类）

以下标准库类将具有其 `Py_TPFLAGS_SEQUENCE` 位集：

- `array.array`
- `collections.deque`
- `list`
- `memoryview`
- `range`
- `tuple`

注意

虽然 `str`、`bytes` 和 `bytearray` 通常被认为是序列，但它们不包含在上面的列表中，也不匹配序列模式。

如果主题序列的长度不等于子模式的数量，固定长度序列模式失败。

如果主题序列的长度小于非星号子模式的数量，可变长度序列模式失败。

主题序列的长度使用内置 `len()` 函数获取（即通过 `__len__` 协议）。但是，解释器可以以类似于值模式描述的方式缓存此值。

固定长度序列模式从左到右将子模式匹配到主题序列的相应项。匹配在子模式失败时立即（失败）停止。如果所有子模式都成功匹配其对应项，则序列模式成功。

可变长度序列模式首先将前导非星号子模式匹配到主题序列的相应项，如同固定长度序列。如果这成功，星号子模式匹配由剩余主题项形成的列表，从末尾移除与星号子模式后面的非星号子模式对应的项。然后将剩余的非星号子模式匹配到相应的主题项，如同固定长度序列。

#### 映射模式

语法：

```
mapping_pattern: '{' [items_pattern] '}'
items_pattern: ','.key_value_pattern+ ','?
key_value_pattern:
    | (literal_pattern | value_pattern) ':' pattern
    | double_star_pattern
double_star_pattern: '**' capture_pattern
```

（注意，`**_` 由此语法禁止。）

映射模式最多可以包含一个双星模式，而且它必须在最后。

映射模式不能包含重复的键值。（如果所有键模式都是字面量模式，则视为语法错误；否则这是运行时错误，将引发 `ValueError`。）

对于映射模式要成功，主题必须是映射，其中映射的定义是其类为以下之一：

- 从 `collections.abc.Mapping` 继承的类
- 已注册为 `collections.abc.Mapping` 的Python类
- 具有 `Py_TPFLAGS_MAPPING` 位集的内置类
- 从上述任何类继承的类（包括在父类的 `Mapping` 注册之前定义的类）

标准库类 `dict` 和 `mappingproxy` 将具有其 `Py_TPFLAGS_MAPPING` 位集。

映射模式成功当且仅当映射模式中给出的每个键都存在于主题映射中，并且每个键的模式与主题映射的相应项匹配。键始终使用 `==` 运算符比较。如果存在 `'**' NAME` 形式，则该名称绑定到包含主题映射中剩余键值对的 `dict`。

如果在映射模式中检测到重复键，则模式无效，并引发 `ValueError`。

使用主题 `get()` 方法的两参数形式来匹配键值对。因此，匹配的键值对必须已经存在于映射中，而不是由 `__missing__` 或 `__getitem__` 即时创建的。例如，`collections.defaultdict` 实例只能由匹配时已存在的键的模式匹配。

#### 类模式

语法：

```
class_pattern:
    | name_or_attr '(' [pattern_arguments ',?'] ')'
pattern_arguments:
    | positional_patterns [',' keyword_patterns]
    | keyword_patterns
positional_patterns: ','.pattern+
keyword_patterns: ','.keyword_pattern+
keyword_pattern: NAME '=' pattern
```

类模式不能重复同一个关键字多次。

如果 `name_or_attr` 不是内置 `type` 的实例，则引发 `TypeError`。

如果主题不是 `name_or_attr` 的实例，则类模式失败。使用 `isinstance()` 测试。

如果没有参数存在，则如果 `isinstance()` 检查成功，模式成功。否则：

- 如果只存在关键字模式，则按如下方式逐一处理：
    - 关键字作为属性在主题上查找。
        - 如果引发 `AttributeError` 以外的异常，则异常向上冒泡。
        - 如果引发 `AttributeError`，则类模式失败。
        - 否则，与关键字关联的子模式与属性值匹配。如果这失败，类模式失败。如果成功，匹配继续到下一个关键字。
    - 如果所有关键字模式都成功，则类模式整体成功。
- 如果存在任何位置模式，则将它们转换为关键字模式（见下文），并作为附加关键字模式处理，在语法关键字模式之前（如果有）。

位置模式使用 `name_or_attr` 指定的类上的 `__match_args__` 属性转换为关键字模式，如下所示：

- 对于以下内置类型（见下文），接受一个位置子模式，它将匹配整个主题。（这里的关键字模式与其他类型相同。）
- 调用等效于 `getattr(cls, "__match_args__", ()))` 的内容。
- 如果引发异常，异常向上冒泡。
- 如果返回值不是元组，则转换失败并引发 `TypeError`。
- 如果位置模式的数量超过 `__match_args__` 的长度（使用 `len()` 获取），则引发 `TypeError`。
- 否则，位置模式 `i` 使用 `__match_args__[i]` 作为关键字转换为关键字模式，前提是后者是字符串；如果不是，则引发 `TypeError`。
- 对于重复的关键字，引发 `TypeError`。

一旦位置模式已转换为关键字模式，匹配就像只有关键字模式一样进行。

如上所述，对于以下内置类型，位置子模式的处理是不同的：`bool`、`bytearray`、`bytes`、`dict`、`float`、`frozenset`、`int`、`list`、`set`、`str` 和 `tuple`。

此行为大致等效于以下代码：

```python
class C:
    __match_args__ = ("__match_self_prop__",)
    @property
    def __match_self_prop__(self):
        return self
```

## 副作用和未定义行为

匹配过程明确产生的唯一副作用是名称绑定。然而，该过程依赖于对主题及其某些组件的属性访问、实例检查、`len()`、相等性和项访问。它还评估值模式和类模式的类名。虽然这些通常不会产生任何副作用，但理论上它们可以。本提案故意省略了对调用哪些方法或调用多少次的所有规范。因此此行为未定义，用户代码不应依赖它。

另一个未定义行为是由捕获模式绑定的变量，这些模式在同一个case块中，后面跟着另一个失败的模式。这些可能早些或晚些发生，取决于实现策略，唯一约束是捕获变量必须在使用它们的守卫明确评估之前设置。如果守卫包含 `and` 子句，只要保持从左到右的评估顺序，操作数的评估可能与模式匹配交错进行。

## 标准库

为方便使用模式匹配，将对标准库进行几项更改：

- 命名元组和数据类将具有自动生成的 `__match_args__`。
- 对于数据类，生成的 `__match_args__` 中属性的顺序将与生成的 `__init__()` 方法中相应参数的顺序相同。这包括从超类继承属性的情况。具有 `init=False` 的字段从 `__match_args__` 中排除。

此外，将进行系统的努力来审查现有标准库类并在看起来有益的地方添加 `__match_args__`。

## 附录A – 完整语法

这是 `match_stmt` 的完整语法。这是 `compound_stmt` 的附加替代项。请记住，`match` 和 `case` 是软关键字，即它们在其他语法上下文中不是保留字（包括在没有预期冒号的行首时）。按照惯例，硬关键字使用单引号，而软关键字使用双引号。

除标准EBNF外使用的其他符号：

- `SEP.RULE+` 是 `RULE (SEP RULE)*` 的简写
- `!RULE` 是否定前瞻断言

```
match_stmt: "match" subject_expr ':' NEWLINE INDENT case_block+ DEDENT
subject_expr:
    | star_named_expression ',' [star_named_expressions]
    | named_expression
case_block: "case" patterns [guard] ':' block
guard: 'if' named_expression

patterns: open_sequence_pattern | pattern
pattern: as_pattern | or_pattern
as_pattern: or_pattern 'as' capture_pattern
or_pattern: '|'.closed_pattern+
closed_pattern:
    | literal_pattern
    | capture_pattern
    | wildcard_pattern
    | value_pattern
    | group_pattern
    | sequence_pattern
    | mapping_pattern
    | class_pattern

literal_pattern:
    | signed_number !('+' | '-')
    | signed_number '+' NUMBER
    | signed_number '-' NUMBER
    | strings
    | 'None'
    | 'True'
    | 'False'
signed_number: NUMBER | '-' NUMBER

capture_pattern: !"_" NAME !('.' | '(' | '=')

wildcard_pattern: "_"

value_pattern: attr !('.' | '(' | '=')
attr: name_or_attr '.' NAME
name_or_attr: attr | NAME

group_pattern: '(' pattern ')'

sequence_pattern:
  | '[' [maybe_sequence_pattern] ']'
  | '(' [open_sequence_pattern] ')'
open_sequence_pattern: maybe_star_pattern ',' [maybe_sequence_pattern]
maybe_sequence_pattern: ','.maybe_star_pattern+ ','?
maybe_star_pattern: star_pattern | pattern
star_pattern: '*' (capture_pattern | wildcard_pattern)

mapping_pattern: '{' [items_pattern] '}'
items_pattern: ','.key_value_pattern+ ','?
key_value_pattern:
    | (literal_pattern | value_pattern) ':' pattern
    | double_star_pattern
double_star_pattern: '**' capture_pattern

class_pattern:
    | name_or_attr '(' [pattern_arguments ',?'] ')'
pattern_arguments:
    | positional_patterns [',' keyword_patterns]
    | keyword_patterns
positional_patterns: ','.pattern+
keyword_patterns: ','.keyword_pattern+
keyword_pattern: NAME '=' pattern
```
